## 一、数据模型评审

### ✅ 已设计合理的部分

文档设计了 6 张核心表，整体思路正确：

| 表名 | 评价 |
|------|------|
| `users` | 基础字段完整，`activation_score` + `register_step` 是关键状态机字段，设计正确 |
| `user_tags` | `tag_type` 枚举设计合理（resource/skill/demand），三步激活数据全存这里，一次性提交对 |
| `opportunities` | 核心字段覆盖，`reward_min/max` 分字段存对，`status` 状态机清晰 |
| `lobster_status` | HP体系独立成表是对的，不要塞进users表 |
| `hp_logs` | 有日志表 ✅，这个不能省 |
| `match_records` | `review_score` 存这里对，但需要补充字段（见下） |

### 🚨 缺失字段与问题

**问题1：`opportunities` 表缺少过期机制**

需要添加：
```sql
expires_at DATETIME    -- 需求有效期
view_count INT          -- 浏览数（首页实时感来源之一）
apply_count INT         -- 已申请人数（"已有12人感兴趣"来自这里）
```

**问题2：`match_records` 缺少关键状态字段**

文档里有"3天无操作→再提醒→过期"的逻辑，但表里没有：
```sql
reminded_at DATETIME         -- 第一次催单时间
expired_reason VARCHAR       -- 过期原因（用户放弃/超时/需求已关闭）
buyer_confirmed_at DATETIME  -- 买方确认完成时间（触发分账）
```

**问题3：好评权重计算缺少依赖字段**

文档定义了三级好评权重（普通/认证/高活跃用户），但 `users` 表没有：
```sql
is_verified TINYINT(1)       -- 是否通过认证
monthly_match_count INT      -- 月度对接次数（判断高活跃用户）
credit_score INT             -- 信用分（绕单惩罚来源）
```

**问题4：`referrals` 表缺字段**

需要追踪奖励发放时间，避免重复发：
```sql
reward_granted_at DATETIME   -- 奖励发放时间（NULL=未发放）
referrer_rewarded_hp INT     -- 实际发放的HP数量（阶梯奖励不同档次不同）
```

**问题5：缺少防刷好评的IP/设备表**

文档要求"同一IP注册账号对，好评不计入"——需要：
```sql
user_devices (user_id, register_ip, device_fingerprint, created_at)
```

## 二、接口设计评审

### ✅ 已定义的10个接口覆盖主干，基本到位

### 🚨 接口设计问题

**问题1：`GET /api/feed/stream` 建议明确用轮询**

- SSE 在国内微信/小程序环境支持不稳定
- 初期数据量小，30秒轮询完全够用
- **建议**：统一用30秒轮询，接口返回最近N条"真实/模拟混合"数据，后期平滑切换

**问题2：`POST /api/lobster/daily-run` 不应该是HTTP接口**

这是定时任务，暴露为 HTTP 接口有安全风险（可以被外部随意触发）。  
**建议**：改为内部 Cron Job，不暴露 HTTP 端点。如果需要手动触发，加 `X-Internal-Token` 头鉴权，且只允许 localhost 调用。

**问题3：缺少关键接口（需补充）**

| 缺失接口 | 说明 |
|---------|------|
| `POST /api/opportunities` | 发布需求（用户发布/运营发布） |
| `POST /api/match/apply` | 申请对接（用户点"立刻匹配"） |
| `GET /api/user/activation` | 查询激活度详情（前端激活度组件用） |
| `POST /api/user/referral-bind` | 注册时绑定邀请码 |
| `GET /api/lobster/daily-opportunities` | 查看龙虾今日为我找到的机会 |
| `POST /api/review/report` | 举报绕单 |

## 三、龙虾HP引擎与定时任务评审

### 设计逻辑整体可行，但有工程细节风险

**问题1：HP 扣减并发安全**

每日凌晨2点所有用户同时 -3 HP，用户量大时会有锁表风险。  
**建议**：用队列（Redis Queue）分批处理，每批200人，间隔10秒。或改为**滚动24小时制**（用户激活后24小时后扣一次），避免流量洪峰。

**问题2：HP耗尽状态判断时机**

HP 耗尽时应即时更新 `lobster_status.status` 字段（active→sleeping），用户登录时直接读该字段，不需要每次实时计算。

**问题3：推送机制未明确技术实现**

- 微信小程序：订阅消息（需用户授权）
- 服务号：模板消息（更好的到达率）
- **建议**：MVP 阶段用微信服务号模板消息或小程序订阅消息，免费且到达率高。

**问题4：龙虾匹配算法未定义**

MVP 建议使用简单规则打分：
```
匹配分 = 技能标签交集数 × 2 + 资源标签交集数 × 1 + 同城市 × 5
取 TOP5 推送
```
不需要机器学习，规则引擎足够，后期数据积累后再优化。

## 四、支付分账与合规评审

### ✅ 方案方向正确（微信分账托管 + 联系方式加密）

### 🚨 具体落地风险

**风险1：微信分账需要公司主体 + 接收方预先绑定**

- 个体工商户不支持微信分账的全部功能
- 资源方（接款方）需要**提前注册为分账接收方**，并绑定微信支付账号或银行卡
- 这个用户绑定流程目前产品设计中未提及，**需要补充设计**

**风险2：联系方式加密托管实现复杂**

MVP 阶段建议简化为：通过平台内聊天沟通，需求确认完成后才显示联系方式，不做复杂加密托管。

**风险3：8%佣金的资金流合规**

- 需确认平台是否需要开增值税发票（企业用户会要求）
- 资金在平台账户停留时间过长有被认定"资金池"的风险
- **建议**：咨询财税律师，明确平台税务身份（代理服务费 vs 居间服务费）

## X、AI编排层：DeerFlow + OpenClaw 技术架构

> **本节是 v5.0 产品"赛博玩法"的核心技术支撑，原报告完全缺失此节。**

### X.1 为什么选 DeerFlow 而非 OpenClaw

| 维度 | OpenClaw（裸跑） | DeerFlow 2.0 |
|------|-----------------|--------------|
| 任务编排 | 简单线性脚本 | 支持分支/条件/循环/人工节点 |
| 状态持久化 | 无，断点全重来 | Checkpoint + 断点续跑 |
| 沙箱隔离 | 无，多任务互相污染 | Docker 沙箱隔离 |
| 内容验证 | 无，AI自由发挥 | 内置验证层 + 人工审核节点 |
| 监控/审计 | 弱日志 | 全链路观测 + 审计 |
| 多租户 | 无 | 支持 |
| **结论** | 玩具/原型阶段 | **企业级生产环境** |

**直接结论**：DeerFlow 是 OpenClaw 的超集，接 DeerFlow 就是接 OpenClaw + 所有企业级能力，不需要两条腿走路。

### X.2 架构分层

```
用户（小程序/H5）
        ↓
  Laravel 平台核心
  ├── 用户体系 / HP引擎 / 撮合分账
  ├── 定时触发 → DeerFlow API
  └── 推送结果 → 微信订阅消息 → 用户
        ↓
  DeerFlow 2.0（AI 任务编排层）
  ├── 赛博资本家 Agent（任务规划 + 策略）
  ├── 数字打工人 Agent（具体执行）
  ├── 验证节点（人工审核 + 内容合规）
  └── 状态持久化（MySQL记录任务状态）
        ↓
  OpenClaw / 浏览器自动化层
  ├── 信息抓取（1688/全网）
  ├── 内容发布（用户审核后）
  └── 微信操作（联系/确认）
```

### X.3 DeerFlow 核心任务流

**龙虾每日自动执行流（DeerFlow 驱动）**：

```
触发时机：每天早上 6:00（Laravel Cron → DeerFlow API）

DeerFlow 工作流：
  Step 1: 赛博资本家分析
    → 读取用户画像（标签 + HP状态 + 历史偏好）
    → 决定今日搜索方向（3-5条精准关键词）

  Step 2: 数字打工人执行
    → OpenClaw 浏览器自动化抓取 1688 / 信息平台
    → 清洗数据、去重、质量打分

  Step 3: 内容验证节点（人工审核）
    → DeerFlow 暂停，等待运营/用户确认
    → 用户点"通过"→ 进入推荐池
    → 用户点"拒绝"→ 记录偏好，继续下一条

  Step 4: 匹配推送
    → DeerFlow 触发 Laravel API
    → Laravel 查询匹配用户 → 推送微信订阅消息
    → 用户收到：🦞 阿旺帮你找到3条新机会

  Step 5: 结果反馈循环
    → 用户点"感兴趣" / "不感兴趣"
    → 反馈数据写回 DeerFlow 记录
    → 下次执行时学习偏好
```

### X.4 DeerFlow 与 Laravel 的集成点

| 方向 | 接口 | 说明 |
|------|------|------|
| Laravel → DeerFlow | `POST /api/deerflow/trigger` | 触发每日龙虾任务，传入用户列表 |
| Laravel → DeerFlow | `POST /api/deerflow/verify` | 人工审核节点回调（用户审核通过/拒绝） |
| DeerFlow → Laravel | `POST /api/match/lobster-push` | 任务完成后推送到 Laravel 推荐队列 |
| DeerFlow → Laravel | `GET /api/user/profile` | 执行前拉取用户画像（标签/HP/偏好） |
| DeerFlow 读取 | MySQL `lobster_status` 表 | 查询用户 HP 和龙虾状态 |
| DeerFlow 写入 | MySQL `lobster_task_logs` 表 | 记录每次执行的详细日志 |

### X.5 OpenClaw 在这个架构里的定位

OpenClaw 是 DeerFlow 的**执行终端**，不再是独立的定时任务。

- DeerFlow 通过 Python SDK 调用 OpenClaw
- OpenClaw 只负责"浏览器操作"（抓取、点击、填写）
- 任务编排、状态管理、错误重试、结果验证全部由 DeerFlow 负责

### X.6 为什么这是 P0 风险（原报告漏掉了）

小龙虾平台的差异化竞争力有两条腿：

| 腿 | 内容 | 技术依赖 |
|----|------|---------|
| **腿1：撮合平台** | Laravel + MySQL + 小程序 | 中等难度，标准技术栈 |
| **腿2：赛博玩法（龙虾自动执行）** | DeerFlow + OpenClaw | **这是产品核心卖点，用户留下来的理由** |

如果只做腿1，你的平台是另一个"列表+筛选"的本地信息服务，零护城河。

如果同时做腿2，用户因为"我的龙虾每天帮我找机会"留下来，有真实粘性。

**原报告的问题**：只评了腿1，把腿2当成"定时任务"简单带过——这是方向性失误。

### X.7 接入 DeerFlow 的实际成本

| 阶段 | 工作量 | 预计时间 |
|------|--------|---------|
| DeerFlow 部署 + Hello World | 1-2天 | 有 Docker 镜像，快速 |
| 与 Laravel 平台对接（API集成） | 1-2周 | 4个核心接口 |
| 龙虾每日执行流开发 | 2-3周 | 最复杂部分 |
| 人工审核节点 UX | 1周 | 用户体验设计 |
| **MVP阶段 DeerFlow 总计** | **约 1 个月** | |

> ⚠️ 如果 MVP 先不做 DeerFlow，赛博玩法在前3个月靠"手动运营"替代（运营人员人工推荐），验证产品价值后再接入 DeerFlow。这是合理的分阶段策略。

### XI.1 合规命名规范（技术开发必须遵循）

### XI.2 用户等级体系（技术团队可直接开发）

**一个核心原则**：用户等级 = 订阅月包档位（只有档位，无"充值余额"）。

| 等级 | 月包名称 | 价格 | 核心能力 | 技术实现 |
|------|---------|------|---------|---------|
| **L0** | 免费用户 | 0元 | 大厅浏览 + 每日5次主动申请 | 仅基础功能，无需DeerFlow |
| **L1** | 基础月包 | 49元/月 | L0全部 + 每日3条被动推送 | Laravel定时任务 + 微信订阅消息 |
| **L2** | 高级月包 | 99元/月 | L1全部 + **龙虾分身开启** + 1688选品工具 + 淘客链接无限生成 | DeerFlow执行流 + 阿里API |
| **L3** | 旗舰月包 | 199元/月 | L2全部 + 三种性格自由切换 + 优先人工审核 + 内容生成无限 | DeerFlow高级流 + 运营人工介入 |

**增值包（一次性购买，非月包，不产生余额）**：

| 增值包 | 价格 | 内容 | 技术实现 |
|--------|------|------|---------|
| 增值包A | 9元 | 额外10次主动申请次数（当月有效） | 字段叠加，不改月包逻辑 |
| 增值包B | 19元 | 额外30天龙虾分身有效期（月包到期后延续） | 叠加 `lobster_expires_at` |
| 增值包C | 29元 | 解锁第三种性格"全网型"（仅L2用户购买） | 性格字段解锁，不需要改DeerFlow |

**月包续费逻辑**：
- 月包到期 → 龙虾分身休眠（HP停止消耗）→ 降级为L1（保留推送）→ 再降为L0（仅大厅）
- HP不清零，续费后立即恢复（HP是用户历史积累，不随月包到期消失）

### XI.3 龙虾分身性格系统（AI推荐策略配置）

> **核心定义**：龙虾分身性格 = AI搜索策略配置，不是金融风险等级，不需要任何金融牌照。

性格系统在 DeerFlow 里体现为**不同的任务流参数模板**，三种性格共用同一套执行引擎，仅参数不同。

| 性格 | 场景定位 | 搜索数据源 | 内容生成方向 | DeerFlow模板参数 |
|------|---------|---------|-----------|----------------|
| **同城型** | 找本地差价、社区流量、线下机会 | 本地分类信息平台 + 小红书同城 + 闲鱼本地 | 社区推广文案、同城转让文案 | 关键词：同城、附近、低价、转让、社区 |
| **电商型** | 有货/有供应链，想通过电商变现 | 1688API + 淘宝联盟 + 拼多多 | 商品种草笔记、电商详情页文案 | 关键词：佣金率>15%、销量上升、差评率<5% |
| **全网型** | 有资源/有粉丝，想全网分发变现 | 全网Affiliate + 多平台内容API | 各平台适配分发文案（小红书/抖音/闲鱼/公众号） | 关键词：全网热门话题、CPS佣金、内容分发 |

**性格解锁规则**：

| 用户等级 | 可用性格 | 能否购买增值包C解锁全网型 |
|---------|---------|----------------------|
| L0/L1 | 无龙虾分身 | 不适用 |
| L2（99元/月） | 同城型 + 电商型 | 可以（29元解锁全网型） |
| L3（199元/月） | 三种性格全开 | 已包含，无需购买 |

**性格切换**：
- 用户可在个人中心随时切换性格
- 切换后，次日凌晨的龙虾任务使用新性格参数
- 不收费，不影响HP

### XI.4 龙虾分身的技术执行流程（L2/L3用户）

以下流程由 DeerFlow 驱动，**所有操作节点用户必须有机会审核和拒绝**：

```
每日早上 6:00 触发（按用户所在城市分组，Redis队列分批）

① 读取用户画像
   - 用户标签（resource/skill/demand）
   - 用户等级（L2/L3）
   - 当前性格（同城/电商/全网）
   - 历史反馈（点过哪些推荐、不点哪些）

② AI搜索 + 筛选（性格模板决定搜索关键词和数据源）
   - 同城型：扫描本地分类信息API + 小红书同城
   - 电商型：1688选品API + 淘客联盟API
   - 全网型：全网Affiliate热度榜单 + 多平台内容API

③ 质量打分 + 去重
   - 过滤重复/低质量/已过期信息
   - 按匹配度排序，取TOP5

④ 生成内容草稿（电商型/全网型）
   - 生成标题 + 摘要 + 小红书/闲鱼文案
   - 不自动发布，只生成

⑤ 推送给用户（必须经过此节点）
   - 微信订阅消息："🦞 阿旺今早帮你找到3条新机会"
   - 包含：标题 + 佣金区间（若有）+ [查看] [不感兴趣]

⑥ 用户操作
   - 点"查看" → 进入机会详情页 → 选择申请/忽略
   - 点"不感兴趣" → 记录偏好，下次调整推荐
   - 3天无操作 → 再推送一次提醒

⑦ 结果反馈
   - 用户最终操作写回 hp_logs
   - HP增减：推送被查看+5HP，完成对接+15HP
```

**合规护栏（技术必须实现）**：
- 步骤②-④的所有数据必须是**公开可获取的合法数据源**
- 步骤④生成的内容**必须标注"AI辅助生成，仅供参考"**
- 步骤⑤-⑥用户拥有**完整否决权**，AI不能代替用户做决定
- 所有分佣信息必须注明"佣金可能变动，以平台实际结算为准"

### XI.6 淘客分佣的技术实现路径

> **v5.0 原文档未提及此收入模型，本次评审补充。**

```
用户（电商型性格）→ AI推荐1688热销商品
→ 生成淘客推广链接（淘宝联盟API）
→ 用户点击链接 → 在淘宝/天猫完成购买
→ 淘宝联盟结算佣金 → 打入平台淘客账户（次月结算）

平台收入：佣金 - 8%用户分佣 = 平台留存
用户可见：商品详情页显示"预估佣金 X 元"（淘客API返回）
用户分佣：可在"我的钱包"查看佣金，累计满Y元可提现至微信零钱
```

**技术注意点**：
- 淘客链接有效期通常15-30天，需记录 `link_generated_at`
- 佣金结算有延迟，用户界面要显示"预计到账时间"
- 用户提现走微信付款到零钱接口（需申请），MVP阶段可简化为"佣金抵扣月包费用"

### XII.1 整体执行架构

```
Laravel Cron（每天 06:00 北京时间）
    │
    ▼
POST /api/deerflow/trigger  ← 触发主工作流
    │
    ▼
DeerFlow Master
    │
    ├── Step 1: 读取 Redis 用户队列（按城市分组，每批50人）
    ├── Step 2: 批量调用 Laravel API 获取用户画像
    ├── Step 3: 过滤有效用户（L2/L3 + 龙虾分身未过期）
    ├── Step 4: 按性格分组（同城/电商/全网）
    │
    ├── Step 5: 并行执行性格子工作流
    │         ├── city → OpenClaw 抓取同城信息
    │         ├── ecommerce → 1688 API + 淘客 + AI文案生成
    │         └── global → 全网热点 + 多平台内容分发
    │
    ├── Step 6: 合并结果 + 质量打分
    │
    ├── Step 7: 【人工审核节点】← 核心合规护栏
    │         暂停等待，72小时超时，不允许跳过
    │
    ├── Step 8: 推送结果到 Laravel（微信订阅消息）
    ├── Step 9: 写回用户偏好（反馈循环）
    └── Step 10: 写入执行日志（审计用）
```

### XII.2 主工作流 YAML（lobster_daily_master）

```yaml
workflow:
  name: "lobster_daily_master"
  description: "小龙虾每日自动执行主工作流"
  version: "1.0"

steps:

  # ── Step 1: 读取 Redis 用户队列 ──────────────
  - name: "fetch_user_queue"
    skill: "lobster_user_queue"
    description: "从 Redis 队列读取今日待执行用户列表（按城市分组）"
    output: "user_list"

  # ── Step 2: 批量读取用户画像 ─────────────────
  - name: "fetch_user_profiles"
    skill: "laravel_api"
    depends_on: "fetch_user_queue"
    input: "{{ user_list }}"
    parallel: true
    max_parallel: 10
    output: "profile_batch"
    tool_calls:
      - tool: "http_get"
        url: "{{ LARAVEL_API_BASE }}/api/user/profiles/batch"
        params:
          user_ids: "{{ user_id_list }}"
          fields: "tags,personality,hp_status,user_level,lobster_expires_at"

  # ── Step 3: 过滤有效用户 ─────────────────────
  - name: "filter_eligible_users"
    skill: "lobster_filter"
    depends_on: "fetch_user_profiles"
    input: "{{ profile_batch }}"
    output: "eligible_users"
    filter_logic: |
      return [u for u in profile_batch
              if u.user_level in ['L2','L3']
              and u.lobster_expires_at > NOW]

  # ── Step 4: 按性格分组 ───────────────────────
  - name: "group_by_personality"
    skill: "lobster_filter"
    depends_on: "filter_eligible_users"
    output: "personality_groups"
    group_logic: |
      groups = {'city': [], 'ecommerce': [], 'global': []}
      for u in eligible_users:
        p = u.personality or 'city'
        if p in groups: groups[p].append(u)
      return groups

  # ── Step 5: 并行执行三种性格子工作流 ─────────
  - name: "run_city_flow"
    skill: "deerflow_subagent"
    depends_on: "group_by_personality"
    input: "{{ personality_groups.city }}"
    subworkflow: "lobster_city_flow"
    output: "city_results"
    enabled: "{{ personality_groups.city | length > 0 }}"

  - name: "run_ecommerce_flow"
    skill: "deerflow_subagent"
    depends_on: "group_by_personality"
    input: "{{ personality_groups.ecommerce }}"
    subworkflow: "lobster_ecom_flow"
    output: "ecom_results"
    enabled: "{{ personality_groups.ecommerce | length > 0 }}"

  - name: "run_global_flow"
    skill: "deerflow_subagent"
    depends_on: "group_by_personality"
    input: "{{ personality_groups.global }}"
    subworkflow: "lobster_global_flow"
    output: "global_results"
    enabled: "{{ personality_groups.global | length > 0 }}"

  # ── Step 6: 合并结果 ─────────────────────────
  - name: "merge_results"
    skill: "lobster_merge"
    depends_on: [run_city_flow, run_ecommerce_flow, run_global_flow]
    output: "merged_recommendations"
    merge_logic: |
      all = []
      for src, results in [('city',city_results),('ecommerce',ecom_results),('global',global_results)]:
        for r in results:
          r['source_personality'] = src
          all.append(r)
      return sorted(all, key=lambda x: x['match_score'], reverse=True)[:20]

  # ── Step 7: 【核心合规护栏】人工审核节点 ─────
  - name: "human_review_gate"
    skill: "human_review"
    depends_on: "merge_results"
    input: "{{ merged_recommendations }}"
    output: "reviewed_recommendations"
    review_config:
      mode: "batch"
      max_items_per_batch: 20
      action_buttons: ["全部通过", "逐条审核", "全部拒绝"]
      auto_timeout_hours: 72
      notification_template: "小龙虾已为您找到 {{ count }} 条新机会，请及时审核"
      skip_allowed: false          # ❌ 不允许跳过，平台合规硬性要求

  # ── Step 8: 推送结果到 Laravel ──────────────
  - name: "push_to_laravel"
    skill: "laravel_api"
    depends_on: "human_review_gate"
    tool_calls:
      - tool: "http_post"
        url: "{{ LARAVEL_API_BASE }}/api/match/lobster-push"
        body:
          recommendations: "{{ reviewed_recommendations }}"
          trigger_date: "{{ trigger_date }}"
        headers:
          Authorization: "Bearer {{ LARAVEL_API_TOKEN }}"

  # ── Step 9: 写回用户偏好（异步，不阻塞）──────
  - name: "update_user_preferences"
    skill: "lobster_feedback"
    depends_on: "push_to_laravel"
    async: true

  # ── Step 10: 写入执行日志 ────────────────────
  - name: "write_execution_log"
    skill: "laravel_api"
    depends_on: [push_to_laravel, update_user_preferences]
    always_run: true               # 无论成功失败都记录

error_handling:
  critical_steps: [human_review_gate, push_to_laravel]  # 失败则整流终止
  non_critical_steps: [update_user_preferences, write_execution_log]

resource_limits:
  max_execution_minutes: 30
  max_cost_per_run: 5.00           # USD，超过暂停
  sandbox_timeout_minutes: 10
```

### XII.3 性格子工作流摘要

#### 电商型工作流（lobster_ecom_flow）

> **输出**：商品卡片 + 淘客链接 + 小红书/闲鱼种草文案草稿

```
Step 1: 生成选品策略
  → 根据用户标签（资源/技能），生成1688关键词

Step 2: 调用 1688 选品 API
  → 筛选：佣金率>5%、销量>100件、价格9.9-199元

Step 3: 生成淘客推广链接
  → 淘宝联盟 API → 领券链接 + 佣金率

Step 4: AI 生成种草文案（用户审核后手动发）
  → 小红书风格：真实分享感，不承诺收益
  → 闲鱼风格：简洁，突出性价比
  → 必须附 disclaimer："佣金可能变动，以平台实际结算为准"

Step 5: 推送内容包给用户审核
  → 包含：商品摘要 + 文案草稿 + 链接 + 预估佣金区间
```

#### 同城型工作流（lobster_city_flow）

> **输出**：机会摘要列表（无佣金，纯信息撮合）

```
Step 1: 生成同城关键词（同城差价/社区拼单/本地服务）
Step 2: OpenClaw 抓取小红书同城 + 本地分类信息（58同城）
Step 3: 清洗数据 + 去重 + 匹配度打分
Step 4: 生成简短摘要推送（用户自己决策，不生成内容）
```

#### 全网型工作流（lobster_global_flow）

> **输出**：多平台适配内容包（小红书/闲鱼/抖音脚本）

```
Step 1: 采集全网热点（小红书/抖音/微博/知乎）
Step 2: 热点关联淘客高佣商品
Step 3: 生成各平台适配文案（各平台风格不同）
Step 4: 汇总推送，用户审核后分发
```

## XIII、注册引导三步——前端交互设计规范（技术团队可直接开发）

> **本节是从产品设计文档中提取的前端交互实现规范，包含完整线框图、交互细节、状态机逻辑。**
> **这是用户体验的核心入口，技术开发必须严格还原设计意图，不得擅自简化或\"优化\"交互流程。**

---

### XIII.1 设计原则（产品为什么要这样设计）

产品设计的核心逻辑不是"让用户填表"，而是：

1. **先给诱饵，再提要求**：用户注册后立刻看到"47个匹配机会"，感受到平台有价值，再引导他完善信息
2. **拆成最小步骤**：每步只问一件事（资源/技能/目标），不要一股脑全弹出来
3. **用激活度替代完整度**：不做"资料完善率"，做"龙虾激活度"——给用户一个故事感（龙虾在等你喂它），驱动持续完善

---

### XIII.2 四段式交互流程

```
注册成功
    │
    ▼
第一段：注册落地页（先给诱饵，不弹表单）
    │
    ▼
第二段：三步激活引导（资源 → 技能 → 目标）
    │
    ▼
第三段：激活成功动画（龙虾孵化）
    │
    ▼
第四段：任务中心 → 持续完善（驱动回头）
```

### XIII.7 数据提交时机（关键！）

**三步激活数据必须一次性提交，不允许逐步提交。**

原因：减少 API 调用次数，避免网络抖动导致中间状态数据丢失，避免用户"跳了一步"后在数据库留下残缺状态。

```javascript
// 前端伪代码
// 本地缓存三步所有选择
const onboardingCache = {
  resources: [],   // 第1步选中的标签
  skills: [],      // 第2步选中的标签
  demands: []      // 第3步选中的标签
};

// 三步全部完成后，一次性提交
async function submitOnboarding() {
  await api.post('/api/user/onboarding', {
    user_id: currentUser.id,
    tags: {
      resource: onboardingCache.resources,  // tag_type = 'resource'
      skill: onboardingCache.skills,         // tag_type = 'skill'
      demand: onboardingCache.demands        // tag_type = 'demand'
    }
  });
}
```

**对应的后端接口**：

```php
// POST /api/user/onboarding
// Body: { user_id, tags: { resource: [], skill: [], demand: [] } }
// 逻辑：
// 1. 写入 user_tags 表（三类标签）
// 2. 更新 users.register_step = 'done'
// 3. 计算并更新 users.activation_score
// 4. 返回 { status: 'ok', activation_score: 60 }
```

## XV、平台结构扩展——合规舞台 & 大人物机制 & 分享裂变

> **本节提取自产品设计文档第六节，包含龙虾能力边界、大人物识别与三级扩散机制、分享裂变设计、灵活用工区+大学生专区。**
> **技术开发必须严格遵守龙虾合规边界，不得私自扩展"龙虾能做的事"。**

---

### XV.1 龙虾能力边界——合规原则（技术强制约束）

**核心原则：龙虾是"执行助理"，不是"决策主体"**

| 合规 ✅ | 违规 ❌ |
|--------|--------|
| 信息处理、撮合推荐、任务自动化 | 代替用户做资金操作、法律行为、持牌业务 |
| 跨平台信息整合（官方API） | 爬取有反爬机制的平台（不稳定，易被封）|
| 帮用户生成内容草稿（用户审核后发布） | 代替用户自动发布内容（平台禁止）|

**已确认放弃的方向**（技术架构不应覆盖）：
- 跨平台信息聚合（各平台反爬机制严格）
- 内容生产（AI工具泛滥，无护城河）

**确认保留的核心舞台（优先级）**：

| 舞台 | 优先级 |
|------|--------|
| 线下撮合（本平台核心） | ⭐⭐⭐⭐⭐ |
| 阿里电商全链路（选品/上架/铺货） | ⭐⭐⭐⭐⭐ |
| 大人物分层服务（风投/专家/品牌商） | ⭐⭐⭐⭐ |
| 淘客佣金（流量变现） | ⭐⭐⭐ |

---

### XV.2 大人物识别与分层服务机制

#### 谁是大人物（识别信号）

**被动信号（系统自动识别）**：
- 资源标签含"投资人脉/基金资源"
- 发布需求金额 > 5000元
- 单次充值大额
- 账号被多人@或推荐

**主动信号（用户自填）**：
- 入驻时选择"我是机构/品牌方"
- 填写公司名称+职位

**认证方式**：用户自申请 + 平台运营主动邀请（双轨并行）

#### 大人物需求的三级扩散机制

**第一级：龙虾精准定向推送（私密，系统主导）**
- 需求发布后，龙虾先在平台内做定向匹配
- 匹配到符合条件的资源方，定向推送（用户不知道需求方身份，点击后需实名+信用门槛）

**第二级：大厅置顶展示（公开，信息脱敏）**

```
┌──────────────────────────────────────┐
│ 🔥 高价值需求                          │
│ 寻城市合伙人 · 广州                    │
│ 行业：本地生活服务  预算：面议（大额）  │
│ 需求方：[认证机构] ⭐⭐⭐⭐⭐            │
│ 已有12人感兴趣                         │
│ [我符合条件，申请对接]                  │
└──────────────────────────────────────┘
```

- **不展示需求方真实身份**，龙虾帮大人物筛选第一轮申请者
- 技术实现：需求方 `is_big_person` = true 时，触发置顶逻辑

**第三级：大人物主动授权广播（付费/消耗积分）**

```
[📢 广播这个需求]
选择扩散渠道：
□ 平台站内全体推送
□ 生成分享卡片（发微信/抖音/朋友圈）
□ 只发给我指定的用户群体
费用：广播一次消耗500积分（或付费）
```

#### 大人物龙虾能力升级（自动触发）

| 普通龙虾 | 大人物龙虾（自动升级） |
|---------|----------------------|
| 每日扫描需求，推给用户 | 每日扫描资源方用户，主动推给大人物 |
| 匹配单条机会 | 批量匹配执行团队 |
| 推送5条机会 | 专属顾问服务（运营人员主动联系） |
| 等用户发现需求 | 龙虾主动帮大人物发布需求，代为筛选第一轮候选人 |

### XV.3 分享裂变机制设计

#### 微信分享卡片（技术生成）

```
┌──────────────────────────────────┐
│  🦞 小龙虾平台 · 今日大单           │
│  📍 广州天河 · 本地服务             │
│  💰 佣金：68元/单，今日已成交9单     │
│  🏷 类型：保洁信息差                │
│  [我也想接这类单 →]                │
│  扫码进平台，激活你的龙虾            │
│  ▓▓▓▓▓ 二维码（含邀请码）▓▓▓▓▓    │
└──────────────────────────────────┘
```

**技术要点**：
- 二维码内嵌分享者邀请码（`referral_code`），有人扫码注册即触发拉新奖励
- 卡片数字必须真实（今日成交X单），不能造假——数据来自数据库实时查询
- 生成方式：后端渲染模板图片 或 前端 Canvas 绘制

#### 分享激励规则

| 分享行为 | 奖励 |
|---------|------|
| 分享一条需求到微信（无论是否有人点） | +3 HP |
| 分享后有人注册并激活 | +21 HP（叠加拉新奖励）|
| 分享后有人完成第一次对接 | 额外 +15 HP |
| 大人物广播需求 | 付费或消耗大额积分，全员通知 |

### XV.4 平台板块结构（技术路由设计）

```
大厅（首页）
├── 今日机会（本地撮合，核心板块）        ← 默认展示
├── 🔥 高价值需求（大人物专区，置顶）      ← 大人物需求专用
├── 💼 灵活用工区（数据标注/企业兼职）     ← 独立入口
└── 🎓 大学生专区（独立板块）             ← 大学生标签用户自动路由
```

**路由逻辑**：
- 用户标签含"大学生/在校生" → 注册后自动展示大学生专区入口
- 普通用户默认展示今日机会
- 大人物认证用户额外看到 🔥 高价值需求专区

**龙虾在这里的作用**：

- 标注任务单量大、标准化，适合 AI 批量推荐

## XVI、完整收费体系——技术实现规范

> **本节提取自产品设计文档第七节，包含完整定价结构、支付流程、绕单防控双保险机制、支付架构演进路线。**
> **支付系统是技术风险最高的模块，必须严格按本节实现，不得擅自简化绕单防控机制。**

---

### XVI.1 普通用户收费结构

#### 免费基础功能（永久免费，作为拉新入口）

| 功能 | 限制 |
|------|------|
| 注册激活龙虾 | 无限 |
| 每日查看撮合机会 | 无限制 |
| 基础HP体系 | 上限50 |
| 发布需求/资源 | 1条 |

#### 增值工具包（一次性购买，非订阅制——合规无风险）

| 工具包 | 定价 | 内容 | 有效期 |
|--------|------|------|--------|
| 加速包 | 29元 | HP上限提升至200，每日推送10条机会 | 30天 |
| 电商包 | 59元 | 解锁阿里1688选品推荐功能 | 30天 |
| 专业包 | 128元 | HP上限500+阿里电商全链路+优先展示 | 30天 |

> **叫"工具包"而非"月包"**——用户买的是明确功能，不是余额，不存在资金托管合规风险。技术层面也不要有"余额"字段。

---

### XVI.2 成交佣金（平台主收入）

```
普通撮合成交：平台抽 8%

阶梯降费（鼓励用户持续走平台）：
  月成交 0-5000元：  8%
  月成交 5000-2万：  6%
  月成交 2万以上：   4%
```

**前端显示规则**：
- 展示佣金比例时不展示具体金额公式
- 卡片显示"平台服务费"（不提"佣金"），后端存 `commission_rate`
- 绝对不能出现"预估收益"、"年化"、"月收益率"等金融词汇

---

### XVI.3 微信/支付宝分账托管——支付流程

```
① 买方扫码付款（微信/支付宝）
② 资金进入平台商户号（微信/支付宝全程托管）
③ 双方在平台内点击"确认服务完成"
④ 平台自动触发分账指令
⑤ 平台留取佣金，剩余自动打给资源方
⑥ 资源方确认收款，信用分 +1
```

**技术实现要点**：
- 微信支付：使用微信支付「合单支付」+「分账」API
- 支付宝：使用「资金授权」+「统一收单资金结算」
- **重点**：分账前必须有双方确认动作（防止虚假成交套取佣金）
- 每笔交易记录 `payment_id`、`split_ratio`、`settlement_status`

---

### XVI.4 大人物——龙虾合伙人计划

**准入机制**：邀请制 + 自主申请，人工审核，不在首页公开展示（隐藏入口）

| 等级 | 价格 | 核心权益 |
|------|------|---------|
| 探索级 | 1980元/半年 | 龙虾猎人模式激活 + 需求置顶 + 专属认证标 + 联系方式加密保护 |
| 合伙级 | 4980元/半年 | 探索级全部 + 广播3次/月 + 1对1运营对接 + 城市数据月报 + 佣金8%→10%优惠 |
| 战略级 | 面议（2-5万/年起步） | 合伙级全部 + 城市独家资源优先权 + 联合品牌露出 + 定制龙虾能力 + 最优佣金费率 |

**技术实现**：
- 大人物订单走独立支付通道（对公转账入口 + 发票申请）
- 战略级订单不在系统内完成支付，由运营人员线下处理
- 大人物身份存储：`users.is_big_person = true`、`users.big_person_level = 'explorer' | 'partner' | 'strategic'`

---

### XVI.5 绕单防控双保险机制（技术强制）

#### 第一保：资金约束（技术层面，硬约束）

- **联系方式加密托管**：对接意向确认前，双方只能通过平台内消息沟通，真实手机号/微信不直接暴露给对方
- **付款后才解锁联系方式**：资源方有强烈动力主动走平台完成交易流程
- 技术实现：`users.contact_locked = true`，解锁条件：`payment.status = 'completed' AND both_confirmed = true`

#### 第二保：信用惩罚（行为层面，软约束）

```
绕单被举报并核实：
- 资源方：永久封号 + HP清零 + 列入平台黑名单
- 需求方：信用分降级 + 限制发布需求权限
- 双方：绕单记录在平台内公示（其他用户可见）
```

**技术实现**：
- `users.is_banned = true`，`users.ban_reason = '绕单'`
- `blacklist` 表记录被封原因，对大人物开放查询接口
- 绕单公示：需求详情页加「绕单记录」标签（当 `publisher_id` 在黑名单时显示）

---

### XVI.6 阿里电商——纯佣金分润（技术实现路径）

```
龙虾向用户推荐商品
→ 用户通过平台链接成交
→ 平台自动获得淘客佣金（阿里联盟官方结算）
```

**合规要点**：
- 用户无需额外付费，平台收入来自淘客返点
- 完全合规，被动收入，无需用户支付资质
- 技术实现：使用淘宝联盟开放平台 API，生成带推广位的链接

### XVI.8 支付架构演进路线

```
阶段一（初创期，现在）：
  微信/支付宝分账托管 + 对公转账
  → 无需支付牌照，快速落地

阶段二（规模化后）：
  接入第三方持牌支付公司（连连支付/汇付天下等）
  → 更灵活的分账规则，支持更复杂场景

阶段三（平台成熟后）：
  与银行洽谈资金监管账户
  → 最高信任背书，适合大额机构交易
```

技术架构设计时需预留扩展性，当前阶段一不支持的功能不要硬编码。

## XVII、案例启发 & 合伙创业功能——内容社区 + 撮合闭环

> **本节提取自产品设计文档第八节，包含龙虾情报站、点子广场、合伙空间、打赏机制及合规边界。**
> **内容社区是平台差异化核心，技术实现需从一开始就做好合规过滤，不能留到后期打补丁。**

---

### XVII.1 龙虾情报站——内容频道

**定位**：运营人员维护的内容频道，3天一更（最低频率要求）

**内容来源**：
- AI工具最新进展（GPT/Claude/Sora等，机会向解读）
- 一人公司/小团队创业案例（头条/公众号等来源摘要+原链）
- 传统互联网生意 × AI重做 = 新机会（编辑视角）
- 平台内真实成交案例（脱敏后展示）

**标准内容卡片格式**：
```
【标题】用AI做XX，他一个月赚了3万
【来源】头条 @XXX · 日期
【核心模式】3句话总结
【复制门槛】★★☆☆☆（低/中/高）
【需要的能力】内容创作 / 选品眼光 / ...
【龙虾能帮到哪】自动选品 / 文案生成 / ...
【点击原文】→ 跳转原始链接（摘要+原链，不全文转载，规避版权风险）
```

**技术要点**：
- 内容由运营后台手动发布（不是UGC，不需要审核流）
- 原文链接必须附上（版权合规要求）
- 复制门槛星级是运营填写的枚举值，不是AI生成

---

### XVII.2 点子广场——UGC内容社区

**面向所有用户开放，UGC内容**

**帖子类型**：
- 我有个点子（分享想法，寻求讨论）
- 我在做这件事（分享进展，吸引关注/协作）
- 我需要XX能力的搭档（合伙招募，能力合伙）
- 我看到一个机会（转发+点评外部案例）

---

### XVII.3 合规边界（硬规则，技术强制过滤）

```
✅ 允许：能力合伙、经验分享、互相学习、点评商业模式
❌ 禁止：资金招募、投资承诺、收益保证、出资分红
❌ 触发词自动屏蔽：投资/融资/分红/保本/月收益XX%/年化

免责声明（每帖自动附加）：
"本帖内容为用户个人分享，不构成投资建议，平台不对合作结果承诺。"
```

**技术实现**：
- 发帖时前端实时关键词扫描（`invest`、`分红`、`保本`、`年化收益率` 等）
- 后端二次过滤，即使前端绕过也拦截
- 触发词库可配置，运营可随时增减
- **核心原则：只允许"能力合伙"，严禁"资金合伙"**——这条线画清楚，内容审核就有标准

---

### XVII.4 合伙撮合——中等深度

**流程**：
```
① 用户发布合伙需求（在点子广场标注"寻找搭档"）
② 平台/龙虾匹配推荐（基于用户技能标签，推荐5-10人）
③ 双方互看简介 → 发起"合伙探讨"申请
④ 申请通过 → 进入"合伙空间"
```

**合伙空间功能（中等深度）**：
```
├── 项目描述（对外公开信息）
├── 任务板（发任务、认领、标记进度）
├── 角色分工（各自负责范围）
└── 进展更新（给关注者看）
```

**任务板示例**：
```
项目：AI英语家教小程序
[张三] 负责：产品设计 + 运营
[李四] 负责：小程序开发

待办任务：
□ 竞品分析（张三，本周）
□ 用户需求访谈3个（张三，下周）
□ 首页原型图（张三，下周）
□ 微信小程序注册（李四，本周）
☑ 确定合伙分工（已完成）
```

### XVII.5 打赏回馈机制（软性引导）

**触发节点**：
- 项目第一笔收入 → 平台推送祝贺 + 温馨提示（可选打赏）
- 月收入超1万 → 平台邀请"上墙"成为案例（荣誉激励）
- 用户主动打赏 → 展示"感恩龙虾"勋章（平台曝光加成）

**为什么不强制抽成**：
- 合伙项目多在平台外执行，无法追踪
- 强制引发反感，平台信任度下降
- 软性引导+成功案例展示，形成良性循环

**技术实现**：
- 打赏走微信/支付宝收款码（无平台托管，纯扫码）
- 打赏后解锁"荣誉合伙人"标签（`users.achievement_tags` 字段）
- 无订单跟踪，无抽成逻辑

**场景**：用户在点子广场/合伙空间标记"需要帮助"时，平台主动介入

```
响应层次：
① 龙虾自动扫描：平台内有没有匹配的闲置资源/能力？
② 运营人员介入：对有潜力的项目主动推送平台资源
③ 大人物通道：高价值项目推送给龙虾合伙人计划用户
```

**本次 v1.5 更新说明**：此前技术评审报告遗漏了产品设计文档第六、七、八节的核心内容，导致技术架构存在重大缺口。本次更新（XIV~XVII）补全了全部缺失章节，技术评审现已与产品设计文档全部8节一一对应。

**Section XIV 是用户体验的入口**，前端团队可直接还原首页设计。

**Section XII 是技术实现的核心蓝本**，包含 DeerFlow YAML + Skills + API Contract + Docker 配置。

**Section XVI（收费体系）是技术风险最高的模块**，必须严格实现绕单防控双保险机制。

**Section XVII（合伙创业）是合规最敏感的模块**，内容过滤系统必须从一开始就做好，不能留到后期。

**最大风险总结**：

1. **合规命名**：所有文案/代码字段禁止出现"充值"/"起投"/"收益"/金融风险等级词汇——这是最容易在开发中被忽视的 P0。
2. **AI执行合规护栏**：`human_review_gate` 节点的 `skip_allowed: false` 是平台存活的底线，不能在开发中被"优化"掉。
3. **DeerFlow + OpenClaw 整合**：Section XII 的任务流是技术实现的核心蓝本，开发团队应以此为准。
4. **支付资质**：微信分账需要公司主体，且资源方需预先绑定。
5. **数据冷启动**：大厅需求数量决定第一批用户的激活转化率，需要提前准备种子数据。

**建议下一步顺序**：
1. **Section XVI（收费体系）交开发团队评审**（支付系统是最高风险模块，需尽早确认技术路径）
2. **Section XVII（合伙创业）合规过滤系统优先实现**（关键词屏蔽必须从MVP就做好，后期加代价极高）
3. 公司主体注册 + 微信支付商户号申请（与开发并行）
4. Section XII DeerFlow Docker 部署 + Hello World（预计2天）
5. Laravel API 接口对接（Section XII 的 4 个接口，预计1周）
6. 运营种子数据录入（100+条真实需求，人工准备）
7. Section XIV 首页开发（前端团队可同步开始）
8. **Section XV 大人物专区**（建议 MVP 后再做，优先级低于核心撮合流程）
